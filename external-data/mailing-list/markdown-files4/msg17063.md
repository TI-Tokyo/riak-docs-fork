&gt;&gt;&gt;&gt; Location(namespace, key))

&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; The thing to keep in mind about read repair is -- it happens
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; asynchronously on every GET, but /after/ the results are returned to 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; the
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; client.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; So, when you issue a GET with r=1, the coordinating node only waits
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; for 1 of the replicas before responding to the client with a success, 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; and
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; only afterwards triggers read-repair.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; It's true that with notfound\_ok=false, it'll wait for the first
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; non-missing replica before responding. But if you edit or update your
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; objects at all, an R=1 still gives you a risk of stale values being
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; returned.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; For example, say you write an object with value A. And let's say
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; your 3 replicas now look like this:
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; replica 1: A, replica 2: A, replica 3: notfound/missing
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; A read with an R=1 and notfound\_ok=false is just fine, here.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; (Chances are, the notfound replica will arrive first, but the 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; notfound\_ok
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; setting will force the coordinator to wait for the first non-empty 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; value,
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; A, and return it to the client. And then trigger read-repair).
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; But what happens if you edit that same object, and give it a new
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; value, B? So, now, there's a chance that your replicas will look like 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; this:
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; replica 1: A, replica 2: B, replica 3: B.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; So now if you do a read with an R=1, there's a chance that replica
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; 1, with the old value of A, will arrive first, and that's the response 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; that
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; will be returned to the client.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; Whereas, using R=2 eliminates that risk -- well, at least decreases
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; it. You still have the issue of -- how does Riak decide whether A or B 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; is
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; the correct value? Are you using causal context/vclocks correctly? 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; (That
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; is, reading the object before you update, to get the correct causal
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; context?) Or are you relying on timestamps? (This is an ok strategy,
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; provided that the edits are sufficiently far apart in time, and you 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; don't
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; have many concurrent edits, AND you're ok with the small risk of
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; occasionally the timestamp being wrong). You can use the following
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; strategies to prevent stale values, in increasing order of
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; security/preference:
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; 1) Use timestamps (and not pass in vector clocks/causal context).
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; This is ok if you're not editing objects, or you're ok with a bit of 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; risk
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; of stale values.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; 2) Use causal context correctly (which means, read-before-you-write
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; -- in fact, the Update operation in the java client does this for you, 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; I
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; think). And if Riak can't determine which version is correct, it will 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; fall
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; back on timestamps.
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt;
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; 3) Turn on siblings, for that bucket or bucket type. That way,
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; Riak will still try to use causal context to decide the right value. 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; But if
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; it can't decide, it will store BOTH values, and give them back to you 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; on
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; the next read, so that your application can decide which is the 
&gt;&gt;&gt;&gt;&gt;&gt;&gt;&gt; correct one.
